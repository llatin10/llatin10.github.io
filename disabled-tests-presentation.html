From Noise to Signal: A Tracked, Self-Enforcing Disabled-Tests Workflow

Broken automation tests were cluttering Allure and getting silently switched off with no record of why. Across two PRs — #1955 and #1960, both now merged — we standardized how tests are disabled, generated a single source-of-truth registry, made CI enforce it, shipped a one-command self-service tool, made the scope auto-adapt to the team's domains, and cleared a security finding.

✨ Key Highlights

🪄 Hands-free (new — #1962): the registry regenerates and commits itself on every PR. You just push your code.

33+ broken tests disabled the agreed way — visible, each with a reason and an OZ ticket.

DISABLED_TESTS.md: one auto-generated registry (domain · test · platform · reason · ticket).

CI guard: a PR fails if the registry drifts from the code — with the exact fix printed.

/disabled-tests: one command to disable or enable a test, regenerate the registry, and sync Confluence.

Scope auto-derived from the TeamBanking tag group in build.gradle.kts — no hardcoded list to maintain.

Security: removed the unused, vulnerable allure-generator dependency (CVE-2026-33166).

The Problem

Allure showed red from tests that were already known to be broken, drowning real failures in noise. People "fixed" it by quietly commenting out @Test — but with no reason, no ticket, and no list, those tests silently vanished and were never revisited. There was no single place to answer the basic question: what is turned off, and why?

#1955 — Conventions + Registry + CI Guard (merged)

Three disable conventions, one per situation. Broken on BOTH platforms: comment out @Test with a reason line above it (this also keeps it out of Allure). Broken on ONE platform: use @DisabledIfEnvironmentVariable so it still runs on the other platform. Data-driven case: remove the CSV row and log it in a sidecar file. A Python generator then scans the code and writes DISABLED_TESTS.md, and the merge CI runs that generator with --check, failing the PR if the committed registry is out of date.

// DISABLED(broken): Needs user-provisioning job - OZ-75134
//    @Test
fun `Savings - Monthly saving - Immediate Partial Withdrawal`() { ... }

#1960 — Self-Service & Automation (merged)

A shared /disabled-tests command is committed into the repo, so every developer who pulls mobile_dev gets it automatically. It disables or enables a test the agreed way, regenerates the registry, and syncs the Confluence tracker — in one step. The generator's scope is now read live from the TeamBanking tag group in build.gradle.kts, so adding or removing a domain there updates the registry scope automatically, with no script edit. We also removed the unused allure-generator dependency to clear CVE-2026-33166.

python3 scripts/disabled_tests_report.py          # regenerate the registry
python3 scripts/disabled_tests_report.py --check   # CI: exit 1 if stale

🪄 Now shipping (#1962) — the registry maintains itself

A new CI workflow (sync-test-registries.yml) regenerates the registries on every PR and, if anything is stale, commits the fix back to your branch automatically — as github-actions[bot]. No script to run, no hook to install. It covers both the disabled-tests registry and the new automation-coverage registry, using the repo-scoped GITHUB_TOKEN (no long-lived secret).

# on every PR: regenerate -> if stale, auto-commit back to the branch
git commit -m "chore: auto-sync test registries from code"   # github-actions[bot]

How the Team Uses It

Just push your code. Disable or re-enable a test and open your PR — the bot regenerates DISABLED_TESTS.md and commits it for you, so you no longer run a script to keep CI green. The --check gate stays as the backstop, so you never merge a stale registry. (Edge case: if your last push leaves it stale and you merge immediately, the green check needs one "Re-run".) Reach for /disabled-tests when you want to sync the Confluence page or apply a convention.

Impact

Allure is clean — only running tests appear. Every disabled test is now visible with a reason and a ticket. The registry can no longer silently drift from the code. New domains are picked up automatically. And there is one fewer vulnerable dependency on the classpath.
