Disabled-Tests Workflow

2026-06-09

A live demo of how broken automation tests are disabled, tracked, and enforced — and how the registries now keep themselves in sync. Built across #1955 / #1960 (merged) plus the coverage registry + hands-free auto-sync in #1962. Suggested run order: (1) show --check passing; (2) toggle a test to show the CI-style failure with the exact fix; (3) ★ push it to a PR and watch the bot auto-commit the corrected registry — the hands-free beat; (4) point at the build-project.yml --check backstop; (5) show the scope is read live from the TeamBanking tag group in build.gradle.kts.

Key changes

33+ broken tests disabled the agreed way — visible, each with a reason and an OZ ticket (#1955).

DISABLED_TESTS.md: one auto-generated registry — domain, test, platform, reason, ticket.

CI guard fails a PR if the registry drifts from the code, and prints the exact one-line fix.

/disabled-tests: one command to disable/enable a test, regenerate the registry, and sync Confluence (#1960).

Scope auto-derived from the TeamBanking tag group in build.gradle.kts — and a security cleanup removing the unused allure-generator (CVE-2026-33166).

Code highlights

1. The three disable conventions

How a test is turned off depends on where it's broken — and each form leaves a visible, traceable trace in code.

// Both platforms -> comment out @Test with a reason above it (also hides it from Allure):
// DISABLED(broken): Needs user-provisioning job - OZ-75134
//    @Test
fun `Savings - Immediate Partial Withdrawal`() { ... }

// One platform -> still runs on the other:
@DisabledIfEnvironmentVariable(named = "platformName", matches = "android") // bug - OZ-85653

// Data-driven -> remove the CSV row and log it in scripts/disabled-tests/csv-removals.csv

2. The CI guard — registry can't silently drift

build-project.yml runs the generator with --check on every PR. Toggle a test without regenerating and the PR fails with a copy-pasteable fix.

❌ DISABLED_TESTS.md is out of date with the test code.
   A test was enabled or disabled without updating the registry.
   Fix: run  python3 scripts/disabled_tests_report.py  and commit DISABLED_TESTS.md.

★ The bot fixes it for you — hands-free (#1962)

Push the stale change to a PR. The sync-test-registries workflow regenerates and commits the corrected registries back to the branch — nobody runs a script. (Live-tested: the bot auto-committed the fix; the --check gate stays as the backstop.)

$ git log --oneline -1 origin/<your-branch>
chore: auto-sync test registries from code   (github-actions[bot])

3. Scope auto-derived from TeamBanking

The generator reads its in-scope domains live from the TeamBanking tag group in build.gradle.kts — the same list that drives CI test routing. Add or remove a domain there and the registry scope follows, no script edit.

// build.gradle.kts — one source of truth for CI routing AND registry scope
"TeamBanking" to listOf("IDM", "Settings", ... , "Onboarding", "Dashboard"),

# disabled_tests_report.py parses it at runtime:
re.search(r'"TeamBanking"\s*to\s*listOf\(([^)]*)\)', build_gradle)
