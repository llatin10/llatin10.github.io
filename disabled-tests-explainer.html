1. Three disable conventions — one per situation

What it does: How a test is turned off depends on where it's broken. Each convention leaves a visible, traceable trace in code instead of a silent deletion.

Code that makes it work:

Both-platform disables also disappear from Allure (commented @Test). One-platform keeps coverage on the working OS. Data-driven removals are logged so they stay visible.

// Both platforms -> comment out @Test with a reason above it:
// DISABLED(broken): Needs user-provisioning job - OZ-75134
//    @Test
fun `Savings - Immediate Partial Withdrawal`() { ... }

// One platform -> still runs on the other:
@DisabledIfEnvironmentVariable(named = "platformName", matches = "android") // bug - OZ-85653

// Data-driven -> remove the CSV row and log it in scripts/disabled-tests/csv-removals.csv

2. The registry generator (DISABLED_TESTS.md)

What it does: A Python script scans all Kotlin test code for every disable mechanism and writes DISABLED_TESTS.md — domain, test, platform, reason, ticket. It is the only thing that builds the file; never hand-edit it.

Code that makes it work:

One source of truth for &#x27;what is turned off, and why&#x27; — always generated, never typed by hand.

python3 scripts/disabled_tests_report.py          # regenerate the registry
python3 scripts/disabled_tests_report.py --check   # exit 1 if stale (used by CI)

3. CI guard — the registry can't silently drift

What it does: build-project.yml runs the generator with --check on every PR. If the committed registry doesn't match the code, the PR fails with a clear, copy-pasteable fix — no tribal knowledge needed.

Code that makes it work:

The failure message tells you exactly what to run — works for any developer, no Claude Code required.

❌ DISABLED_TESTS.md is out of date with the test code.
   A test was enabled or disabled without updating the registry.
   Fix: run  python3 scripts/disabled_tests_report.py  and commit DISABLED_TESTS.md.

4. The shared /disabled-tests command (#1960)

What it does: Committed into the repo at .claude/commands/, so every developer who pulls mobile_dev gets it automatically. It applies the right convention, regenerates the registry, and syncs the Confluence tracker — in one step.

Code that makes it work:

The registry half needs only python3; the Confluence push uses the Atlassian connection (documented as a per-dev prerequisite).

/disabled-tests disable SavingsFooTest on android
/disabled-tests sync          # regenerate + push Confluence

5. Auto-scoped to the TeamBanking tag group

What it does: The generator now reads its in-scope domains live from the TeamBanking tag group in build.gradle.kts — the same list that drives CI test routing. Add or remove a domain there and the registry scope follows, with no script edit. (Dashboard was moved TeamJarvis -> TeamBanking as part of this.)

Code that makes it work:

No more hardcoded domain list to keep in sync — the build config is the one place that defines the team&#x27;s domains.

// build.gradle.kts — single source of truth for CI routing AND registry scope
"TeamBanking" to listOf("IDM", "Settings", ... , "Onboarding", "Dashboard"),

# disabled_tests_report.py parses it at runtime:
re.search(r'"TeamBanking"\s*to\s*listOf\(([^)]*)\)', build_gradle)

6. Security cleanup — allure-generator removed (CVE-2026-33166)

What it does: Wiz flagged allure-generator <2.38.0. The fixed version (2.38.0) transitively forces Kotlin 2.1.x, incompatible with the project's Kotlin 1.9 — and the library was unused in source. Reports are built by the Allure Gradle plugin + allurectl/TestOps, not this dependency. So we removed it: CVE eliminated, no version conflict, lighter classpath.

Code that makes it work:

Verified on CI: tests still write results and upload to Allure TestOps, report unaffected. The adapters (allure-junit5, allure-java-commons, allure-attachments) stay at 2.34.0.

-  implementation("io.qameta.allure:allure-generator:$allureVersion")
+  // removed: unused in source; Allure plugin + allurectl build the report

7. The registries now fix themselves — hands-free (#1962)

What it does: You no longer act on the red --check. A new workflow regenerates the registries on every PR and commits the fix back to your branch for you — the "run the generator + commit" step becomes automatic.

Code that makes it work:

One workflow covers both registries; repo-scoped GITHUB_TOKEN, no long-lived secret. The --check gate stays as the backstop, so a stale registry is never merged.

# .github/workflows/sync-test-registries.yml — on every PR:
python3 scripts/disabled_tests_report.py
python3 scripts/automation_coverage_report.py
# if a registry changed -> commit it back, as github-actions[bot]
git commit -m "chore: auto-sync test registries from code" && git push
