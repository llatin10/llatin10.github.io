📱 Mobile Automation
Knowledge Base

A plain-English overview of how the mobile automation framework works — how to run tests, name UI elements, manage test data, and keep Android PRs safe.

⚙️ Configuration & Runs
⚙️

How to configure a test run

Every run is controlled by a set of environment variables. The most important ones tell the framework where to run, which app version to test, and what kind of run it is.

You can set these in a local capabilities.local.properties file (gitignored) or pass them as CI environment variables. Here are the key ones to know:

📱 platformName — android / ios ☁️ machine — local / browserstack 🌍 env — uat / sit 🏷️ runType — develop / release / mock… 📦 appVersion — last-version or e.g. 1.3.4 🌐 locale — he-IL / IT 📅 eventName — local / manual / nightly
💡 Tip: For local runs, set machine=local and localAppVersion=3.x.x. The framework will use your connected device or emulator instead of BrowserStack.
🏃

Run profiles (runType)

The runType tells the system which profile to use. Each team has its own, and mock runs need a special environment.

  • -develop — nightly against develop branch
  • -release — nightly against release branch
  • -mock — stub BFF, requires env=sit
  • -teamBanking / -teamCustomer… — per-team profiles
💡 The BrowserStack build name is auto-generated as: Android-develop-uat-1.3.4 [nightly] 26-05-2026
🔌

Local vs. BrowserStack

Set machine=local to run on your own device or emulator. Set machine=browserstack to run on a real cloud device (used by CI).

  • Local runs use localAppVersion to pick the build
  • BrowserStack runs need bsUserName + bsAccessKey from env
  • Copy capabilities.local.properties.example to get started locally
🔍 UI Locators
🔍

How elements are named

Every UI element gets a unique ID that reads like a sentence — it tells you the screen, where in a list it is (if any), and what type of element it is.

  • Simple screen: screen_context + element_type
  • Inside a list: screen + row_N + element_type

For example: savings_rules_row_10_toggle means "the toggle in row 10 of the savings rules screen".

🏷️

Element type suffixes

Each element type has a standard suffix. Both Android and iOS use the same IDs.

_button _text _image _toggle _text_field _check_box _radio _scrollable _chip_N _primary_button _secondary_button _loader / _progress
⚠️

The package prefix trap (Android)

This is one of the most common causes of locator failures. The rule is simple but easy to forget:

✅ @AndroidFindBy(id = "my_button") — no prefix needed ❌ @AndroidFindBy(id = "il.co.firstdigitalbank.uat:id/my_button") — wrong!

When writing XPath queries, it's the opposite — you must include the full package prefix in @resource-id.

🚨 Rule: @AndroidFindBy(id = ...) → short ID only. XPath @resource-id → full package prefix always.
📊 Test Data & Users
📊

How test data works

Each feature area has a CSV file under testData/<feature>/. Every row is one test scenario, with a unique testID and testName plus whatever fields that feature needs.

Inside the test, you just access testData.fieldName — no hardcoded values anywhere.

💡 No test data needed? Set the data model type to Unit and skip the CSV entirely.
👤

Available test user types

The User Generator V3 pool provides ready-made users for every scenario. One user per test — no two tests share the same user at the same time.

🏦 Regular users with cards 📈 Securities (portfolio + orders) 💵 FCY (USD accounts) 💳 Card / OD limits 👨‍👩‍👧 Joint accounts 💰 Savings 📋 Cheques (multiple states) 🔄 Onboarding (mid-flow users)
🛠️

Creating a custom test user

When the standard pool doesn't have exactly what you need, you can provision a custom user and lock it for your test. It lives in DashboardTestData.csv.

  1. Clone an existing row in DashboardTestData.csv and give it a unique testId and reservationKey.
  2. Use the User BFF Swagger (/user/getByParameters) to find a real user matching your needs and copy the returned IDs into the CSV.
  3. Validate the user's balance via getBalanceByPortfolio in the Dev Tools Swagger.
  4. Reserve the user via /reserveUsers so no other test grabs it during your test run.
💡 Updating? Never create a new row — find the existing testId and update it in place.
🤖 CI & PR Flows
🏷️

Running automation on an Android PR

Android developers can trigger mobile automation tests directly from their PR — no manual test kick-off needed.

  1. Open a PR in android-banking.
  2. Add one or more domain labels (e.g. Cheques, Savings, Cards).
  3. Add Ready For Review With Automation instead of the normal review label.
  4. After approval, Bitrise builds the APK, uploads it to BrowserStack, runs the matching tests, and posts results back on the PR.
💡 Using just Ready For Review skips automation and runs only normal CI.
🛡️

Locator guard on Android PRs

Every Android PR that changes UI element IDs is automatically scanned to see if any mobile-tests automation would break.

  • Removed locator — may break existing tests
  • Renamed locator — old name may still be in tests
  • Duplicate locator — same ID used twice (Error = block merge)
  • Added locator — informational only

Results are posted as a PR comment. Errors must be fixed before merge; warnings need a review.

📄

This doc stays in sync automatically

AUTOMATION_KNOWLEDGE.md lives in the mobile-tests repo alongside the code. A CI check runs on every PR that touches the config source files — it reads the actual Kotlin code, extracts all environment variable names and RunType values, and fails the PR if any are missing from the doc.

This was set up after discovering that the original Confluence page had multiple stale facts — wrong variable names, missing enum values, and a hardcoded environment name that was actually a variable. The lesson: docs that live in the repo get reviewed like code.

💡 In fact, the CI script caught a missing entry (stubLocalPassword) on its very first run — before it was even committed.
