🧹 From Noise to Signal — the Disabled-Tests Workflow

PRs #1955 + #1960. Broken tests were cluttering Allure and getting silently switched off with no record of why. We standardized how tests are disabled, generated a single source-of-truth registry, and made CI enforce it.

⚠️ The problem: Allure showed red from already-known-broken tests, drowning real failures in noise. People "fixed" it by quietly commenting out @Test — no reason, no ticket, no list. There was no single place to answer: what is turned off, and why?

⚙️ How It Works — a code-level walkthrough

PRs #1955 + #1960. The mechanics QA and automation engineers actually touch.

1. Three disable conventions — one per situation

Each convention leaves a visible, traceable trace in code instead of a silent deletion. Both-platform disables also disappear from Allure; one-platform keeps coverage on the working OS; data-driven removals are logged so they stay visible.

// Both platforms -> comment out @Test with a reason above it (also hides it from Allure):
// DISABLED(broken): Needs user-provisioning job - OZ-75134
//    @Test
fun `Savings - Immediate Partial Withdrawal`() { ... }

// One platform -> still runs on the other:
@DisabledIfEnvironmentVariable(named = "platformName", matches = "android") // bug - OZ-85653

// Data-driven -> remove the CSV row and log it in scripts/disabled-tests/csv-removals.csv

2. The registry generator (DISABLED_TESTS.md)

A Python script scans all Kotlin test code for every disable mechanism and writes the registry — domain, test, platform, reason, ticket. It is the only thing that builds the file; never hand-edit it.

python3 scripts/disabled_tests_report.py          # regenerate the registry
python3 scripts/disabled_tests_report.py --check   # exit 1 if stale (used by CI)

3. CI guard — the registry can't silently drift

build-project.yml runs the generator with --check on every PR. If the committed registry doesn't match the code, the PR fails with a clear, copy-pasteable fix — no tribal knowledge needed.

❌ DISABLED_TESTS.md is out of date with the test code.
   A test was enabled or disabled without updating the registry.
   Fix: run  python3 scripts/disabled_tests_report.py  and commit DISABLED_TESTS.md.

4. The shared /disabled-tests command (#1960)

Committed into the repo at .claude/commands/, so every developer who pulls mobile_dev gets it automatically. It applies the right convention, regenerates the registry, and syncs the Confluence tracker — in one step.

/disabled-tests disable SavingsFooTest on android
/disabled-tests sync          # regenerate + push Confluence

5. Auto-scoped to the TeamBanking tag group

The generator reads its in-scope domains live from the TeamBanking tag group in build.gradle.kts — the same list that drives CI test routing. Add or remove a domain there and the registry scope follows, with no script edit.

// build.gradle.kts — one source of truth for CI routing AND registry scope
"TeamBanking" to listOf("IDM", "Settings", ... , "Onboarding", "Dashboard"),

# disabled_tests_report.py parses it at runtime:
re.search(r'"TeamBanking"\s*to\s*listOf\(([^)]*)\)', build_gradle)

6. Security cleanup — allure-generator removed (CVE-2026-33166)

Wiz flagged allure-generator <2.38.0. The fixed version transitively forces Kotlin 2.1.x, incompatible with the project's Kotlin 1.9 — and the library was unused in source. Reports are built by the Allure Gradle plugin + allurectl/TestOps, not this dependency. Removing it eliminated the CVE with no version conflict.

-  implementation("io.qameta.allure:allure-generator:$allureVersion")
+  // removed: unused in source; Allure plugin + allurectl build the report

What actually changed (the values + flows)

The re-enabled tests drove real framework fixes — the kind of locator/flow drift the registries are meant to surface:

  1. By.id testTag resolution — CustomAndroidDriver now resolves a bare Compose testTag against both exposed forms (with/without the pkg:id/ prefix), exact-match, and escapes apostrophes in xpath fallbacks (e.g. savings_withdrawal's_type_…).
  2. Lobby total from content-desc — SavingsLobbyPage reads the savings total from content-desc (async-loaded), matching the saving card by name first, then category.
  3. Compose summary drift — summary row ids gained a double underscore (row_{n}__description_text) and the amount moved into withdrawal_summary_summary_title_text; repaired in the monthly + daily summary pages.
  4. Skip-guards, not hard-fails — assumeFutureWithdrawalAvailable(), assumeOnExitDay(), and assumeAfterBusinessHours() make tests skip on the wrong day-type or after business hours (Sun–Thu 08:30–18:30 IDT) instead of failing.
  5. Self-healing accounts — each test runs on its own dedicated CUSTOM or RESERVED account, with a pre-flow cleanup that cancels any leftover scheduled movement from a crashed run.

✅ Team Takeaway

👉 How it stays honest: both registries are generated from the code, not hand-maintained, and each has a CI gate that fails a PR if the checked-in file drifts. When you disable/enable a test or add automation, just run /disabled-tests or /automation-coverage and commit the regenerated file — CI reminds you if you forget, and the matching Confluence page is updated in the same step.
