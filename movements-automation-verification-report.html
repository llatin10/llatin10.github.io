P0 — must have

There must be a positive counterpart proving it shows in the default lobby. OZ-76330 already automates this — add it to the regression list rather than writing new.
  2. "Show All" / load-more on the default lobby. Pagination is tested inside the filter (50/scroll) but never on the main lobby. Your own tracking ticket references "Show All historic movements does not crash" (PR #1928) — there is no regression test guarding it. Add a Show-All / pagination-on-lobby test.
  3. Excel export — invalid & boundary date ranges. OZ-70384/70385 only cover the happy path. FCY already has dedicated tests for: custom valid range = 1 year max (OZ-85011), cancel date picker (OZ-85012), file-naming convention (OZ-85013). Movements export needs the same trio: From > To rejected, >1-year range rejected, cancel picker, file-name format.
  4. Movement detail / extra-details per transaction type. OZ-70386 step 8 only spot-checks "extra details appear" inside the filter. The six "Verify X movement" cases are meant to be the per-type detail checks but are screenshot-only. At minimum automate deposit, payment, FCY, mandate extra-details once their TTM steps are written. (MOB-T9813 digital-wallet detail test exists but is disabled — re-enable as part of this.)

  1. Empty state of the whole movements lobby (new account / zero transactions) — only per-chip empty state is covered (OZ-70392 step 4), not the lobby itself.

  3. Pull-to-refresh on the default lobby — tested under filter (OZ-70386 step 11) but not on the main lobby.
