Test Data & Parallelism

The suite runs test classes in parallel (build.gradle.kts: parallel.enabled=true, classes.default=concurrent, fixed.parallelism=11); methods within a class run serially. All Movements tests here are read-only, so there's no data-mutation conflict — the only parallel risk is shared real-user session collisions (single-session / OTP / token invalidation).

The framework's Stubber overrides GraphQL responses (GET_MOVEMENTS overrideWith StubValues(...)) and requires a MOCK user — so stubbed tests are inherently deterministic and parallel-safe.

Ticket Recommended data approach
OZ-70386 (filtering) MOCK + stub GET_MOVEMENTS: fixed multi-category dataset (some chips populated, ≥1 empty)
OZ-60000 (pagination / Show-All) MOCK + stub GET_MOVEMENTS: >50 movements (several hundred) to cross page boundaries
OZ-59998 (extra-details, all types) Mix: stub hard types (CC billing) via MOCK; dedicated real rich user where realism needed — never shared across parallel classes
OZ-59999 (empty state) MOCK + stub empty GET_MOVEMENTS (or dedicated fresh real user)
OZ-70385 (excel export) MOCK + stub known dataset (in-range + empty-range); date-range edges are UI-level

Rule of thumb: prefer MOCK + stub for deterministic data shapes; reserve dedicated real users only where realism matters, and never share one real login across parallel test classes.
